## Challenges And Friction

Amelia (Developer): "The repeated challenge was evidence drift. Several things were architecturally intended, but not yet proven at the exact layer the story cared about."

Murat (Test Architect): "Story 6.1 initially had review findings around weak assertions: one duplicate outbox identity test accepted any PostgreSQL error, and success tests did not verify the durable payload. That tells us our review bar caught a real risk."

Winston (System Architect): "Story 6.2 exposed a deeper consistency issue: RabbitMQ and Service Bus did not share the same durable receive handoff semantics. That was not an individual mistake; it was a cross-adapter parity gap."

Paige (Technical Writer): "Story 6.3 showed docs lagging code. Once Service Bus moved to durable incoming inbox ingestion, several docs still described the previous behavior."

Amelia (Developer): "The resolver script also failed in this retrospective environment because Python could not import `json`. The workflow fallback worked, but the toolchain issue should be treated as operational friction."

## Key Patterns

### Pattern 1: PostgreSQL Is The Proof Boundary

All three stories reinforced that EF InMemory is useful only for mapping, change tracking, or package-local orchestration tests. Relational claims about transactions, uniqueness, locking, claiming, retry, stale state, and terminal evidence belong in PostgreSQL-backed integration tests.

Murat (Test Architect): "This is the strongest lesson from the epic. If the behavior depends on relational semantics, the proof needs the relational provider."

### Pattern 2: Runtime Changes Stayed Narrow

Story 6.1 remained tests-only. Story 6.2 made the smallest runtime change needed for Service Bus durable inbox parity. Story 6.3 made no runtime changes after the inventory found the provider-backed proof already existed.

Winston (System Architect): "That is exactly the architectural posture we want: prove first, patch only the gap, and keep package boundaries intact."

### Pattern 3: Documentation Must Track Behavioral Changes Immediately

Story 6.2 changed Service Bus receive semantics. Story 6.3 found and corrected stale consumer guidance.

Paige (Technical Writer): "Docs are part of the runtime contract here. If setup guidance says the wrong receive path, consumers will build the wrong mental model."

### Pattern 4: Review Findings Improved Evidence Quality

Each Epic 6 story had review findings that were patched before closure. The recurring theme was not broad design failure; it was tightening proof, wording, and configuration validation.

Amelia (Developer): "That is healthy friction. The reviews made the epic stronger without expanding scope sideways."
